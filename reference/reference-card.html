Learn BMAD — Reference Card

The whole method at a glance. BMAD = Build More, Architect Dreams.

The Five Beliefs

1. Plan before you build. A little thinking up front beats a lot of rework.
2. Make decisions explicit. Written down = everyone builds the same thing.
3. Scale rigor to the change. A typo ≠ a platform.
4. Use specialists, not one generalist. A crew beats one AI in every hat.
5. Human in the loop where it matters. Spend attention on the big calls.

The antidote to "vibe coding": one vague chat, winging it.

The Four Phases

1 · AnalysisUnderstand the problem (optional, sharpening)
2 · PlanningWhat & why? → a PRD
3 · SolutioningHow? + units of work → architecture, epics, stories
4 · ImplementationBuild it, one story at a time

Run each workflow in a fresh chat. You don't always use all four.

Right-Sizing

Small changequick-dev (clarify→plan→implement→review)
Major changeFull four-phase method

Quick-dev: compress intent into one clear goal, take the smallest safe path, bring the human in only at clarify / spec-approve / final-review. Over-processing small work is itself a mistake.

The Build Cycle (Phase 4)

Once: sprint-planning → sprint-status.yaml
Per story: create-story → dev-story → code-review
Per epic: retrospective

Each step in a fresh chat. Dev Loop Automation = on the roadmap.
